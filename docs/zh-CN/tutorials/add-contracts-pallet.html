添加一个模块

Substrate Node Template 提供了一个最小可运行的 runtime，你可使用快速打造自定义的区块链 However, in the attempts to remain minimal, it does not include most of the pallets from FRAME.

本指南将向您展示您如何可以将 Contracts模块 添加到您的runtime，以便允许您的区块链支持 Wasm 智能合同。 你可按照类似的方式将其他 FRAME 模块添加到 runtime 里，但需要注意，每个模块在具体配置设定都会略有差异。

安装 Node Template

当你完成创建你第一条 Substrate 链教程时，你的计算机上应该已编译了v2.0.0版本的Substrate Node Template。 如果您尚未完成，请完成该教程。

有经验的开发者可以跳过该教程，您可以参考readme里的步骤说明来安装 。 。

文件结构

我们现在可以修改 substrate-node-template 添加Contracts pallet。

在您最喜欢的代码编辑器中打开 substrate-node-template。 我们将编辑两个文件：runtime/src/lib.rs, 和 runtime/Cargo.toml。

substrate-node-template
|
+-- runtime
|   |
|   +-- Cargo.toml   <-- One change in this file
|   |
|   +-- build.rs
|   |
|   +-- src
|      |
|      +-- lib.rs   <-- Most changes in this file
|
+-- pallets
|
+-- scripts
|
+-- node            <-- changes in this directory
|
+-- ...

导入 Pallet Crate

要添加合约模块，首先应把pallet-contracts 导入到runtime的 Cargo.toml 文件中。 如果您需要Gargo相关参考资料的入门导读，可查询 它的官方文档

打开 substrate-node-template/runtime/Cargo.toml 文件，您将看到 runtime 的所有依赖组件。 例如，它取决于余额模块:

runtime/Cargo.toml

[dependencies]
#--snip--
pallet-balances = { default-features = false, version = '2.0.0' }

Crate特性

导入pallet crates时需要注意的一件事是确保正确设置crate里的 features内容。 在上面的代码中，你会注意到我们设置了default_features = false。 如果您更仔细地浏览Cargo.toml 文件，则会发现类似以下内容：

runtime/Cargo.toml

[features]
default = ['std']
std = [
    'codec/std',
    'frame-executive/std',
    'frame-support/std',
    'frame-system/std',
    'frame-system-rpc-runtime-api/std',
    'pallet-balances/std',
    #--snip--
]

第二行代码将runtime crate的 default 特性定义为 std。 您可以想象，每个pallet crate都有一个相似的配置定义该crate的默认特性。 每个crate的特性则决定了其下游依赖项的特性。 例如，上面代码应解读为：

此Substrate runtime的默认特性为std。 当runtime启用 std 特性时, parity-scale-codec, primitives, client和所有其他列出的依赖项也将设置为 std。

这样使Substrate runtime 既能编译成支持Rust (支持:std) 和 Wasm (不支持: no_std).

要查看这些功能如何在运行时代码中实际使用，我们可以打开项目文件：

runtime/src/lib.rs

//! The Substrate Node Template runtime. This can be compiled with `#[no_std]`, ready for Wasm.

#![cfg_attr(not(feature = "std"), no_std)]
// `construct_runtime!` does a lot of recursion and requires us to increase the limit to 256.
#![recursion_limit="256"]

// Make the WASM binary available.
#[cfg(feature = "std")]
include!(concat!(env!("OUT_DIR"), "/wasm_binary.rs"));

// --snip--

文件头部定义了当我们 不 使用 std 特性的时候，no_std 特性将会被启用。 再往下走几行代码，可以看见在 wasm_binary.rs 导入代码行上面的 #[cfg(feature = "std")] 标志，它表示仅在启用了 std 特性后我们才能导入 WASM 二进制文件。

导入合约模块

既然我们已经对模块的特性有基本认识了，现在就可以开始动手导入合约模块啦。 合同模块可能是FRAME中最复杂的， 所以它提供了一个 的良好例子，说明在添加额外的模块时可能会涉及的一些棘手问题。

首先，我们将通过简单地复制现有模块并更改里面的值来添加新的依赖关系。 因此，根据上面 balances 导入样式， contracts导入将看起来像:

runtime/Cargo.toml

[dependencies]
#--snip--
pallet-contracts = { version = '2.0.0', default_features = false }
pallet-contracts-primitives = { version = '2.0.0', default_features = false }

与其他钱包一样，Contracts pallet 也具有 std 的特性。 当 runtime 是使用其自身的 std 特性构建时，Nicks pallet 也应该使用它的 std 功能来构建。 将以下代码添加到runtime's的 std 功能。

runtime/Cargo.toml

[features]
default = ['std']
std = [
    #--snip--
    'pallet-contracts/std',
    'pallet-contracts-primitives/std',
    #--snip--
]

现在是一个很好的时候来检查到目前为止所有的编译是否正确：

SKIP_WASM_BUILD=1 cargo check

添加合约模块

现在，我们已成功导入智能合约pallet rust包，我们需要将它添加到我们的Runtime。

实现合约接口

每个pallet都有一个称为 Trait的runtime必须实现。

要想明确我们需要为模块实现什么，你可以查看 pallet_contracts::Trait 文档 对于我们的runtime，执行将看起来像这样：

runtime/src/lib.rs

impl pallet_timestamp::Trait for Runtime {

我们将使用 type DetermineContractAddress 作为一个例子来深入了解更多细节， the DetermineContractAddress 文档 需要 ContractAddressFor特性。 Contracts 模块本身已经实现了 pallet_contracts::SimpleAddressDeterminator特性, 因此我们可以使用该实现来满足我们的 pallet_contracts::Trait. 在这一点上，如果你不太理解或想深入了解，建议探索Contracts pallet source code

将合约添加到 construct_runtime! 宏

接下来，我们需要将pallet添加到construct_runtime!宏里面。 为此我们应先确定pallet要暴露的类型，以便让runtime得知这些类型的存在。 可选类型的完整列表可在construct_runtime! 宏的文档中查找到。

如果我们仔细查看合约钱包，我们知道它具有：

  • Storage模块：因为它使用了decl_storage! 宏。
  • Event模块：因为它使用了decl_event! 宏。
  • Callable 函数：因为它在 decl_module!宏中定义了可调用函数。
  • 配置持久化：因为 decl_storage！ 宏有 config() 参数。
  • 在decl_module!宏里定义的Module类型。

因此当我们添加pallet时，它看起来就像：

runtime/src/lib.rs

construct_runtime!(
    pub enum Runtime where
        Block = Block,
        NodeBlock = opaque::Block,
        UncheckedExtrinsic = UncheckedExtrinsic
    {
        /* --snip-- */

        /*** Add This Line ***/
        Contracts: pallet_contracts::{Module, Call, Config, Storage, Event<T>},
    }
);

请注意，并非所有的pallet都会暴露这些runtime类型，有些可能会暴露更多！ 您总是可以查看钱包的源代码或钱包的文档来确定您需要暴露的类型。

这是另一个很好的时间来检查您的runtime到目前为止是否正确地编译。 虽然runtime可以编译，但整个节点不会通过。 所以我们将使用此命令来检查runtime。

SKIP_WASM_BUILD=1 cargo check -p node-template-runtime

暴露合约API

包括合约pallet在内的pallets，暴露自定义runtime的APIs和RPC端点。 在合约pallet例子，能够从链外读取合约状态。

它不需要让合约pallet上的 RPC 调用来我们的链上使用它。 然而，在没有发起交易的情况下我们将会调用节点存储。

我们首先在我们的 Cargo.toml 中添加所需的 API 依赖关系。

runtime/Cargo.toml

[dependencies]
#--snip--
pallet-contracts-rpc-runtime-api = { version = '0.8.0', default-features = false }

runtime/Cargo.toml

[features]
default = ['std']
std = [
    #--snip--
    'pallet-contracts-rpc-runtime-api/std',
]

为获取合约变量状态，我们必须调用一个getter函数返回一个ContractExecResult ，对当前执行状态的封装。

我们需要将返回类型添加到runtime。 将此添加到另一个 use 语句。

runtime/src/lib.rs

/*** Add This Line ***/
use pallet_contracts_rpc_runtime_api::ContractExecResult;
/* --snip-- */

我们现在准备实现合约的运行时 API。 这种情况发生在 impl_runtime_apis！宏靠近您的运行时间结束。

impl_runtime_apis! {
   /* --snip-- */

   /*** Add This Block ***/
    impl pallet_contracts_rpc_runtime_api::ContractsApi<Block, AccountId, Balance, BlockNumber>
        for Runtime
    {
        fn call(
            origin: AccountId,
            dest: AccountId,
            value: Balance,
            gas_limit: u64,
            input_data: Vec<u8>,
        ) -> ContractExecResult {
            let (exec_result, gas_consumed) =
                Contracts::bare_call(origin, dest.into(), value, gas_limit, input_data);
            match exec_result {
                Ok(v) => ContractExecResult::Success {
                    flags: v.flags.bits(),
                    data: v.data,
                    gas_consumed: gas_consumed,
                },
                Err(_) => ContractExecResult::Error,
            }
        }

        fn get_storage(
            address: AccountId,
            key: [u8; 32],
        ) -> pallet_contracts_primitives::GetStorageResult {
            Contracts::get_storage(address, key)
        }

        fn rent_projection(
            address: AccountId,
        ) -> pallet_contracts_primitives::RentProjectionResult<BlockNumber> {
            Contracts::rent_projection(address)
        }
    }
   /*** End Added Block ***/
}

这是另一个很好的时间来检查您的runtime到目前为止是否正确地编译。

SKIP_WASM_BUILD=1 cargo check -p node-template-runtime

正在更新外节点

此时，我们已经完成添加一个pallet到runtime。 我们现在将注意力转向外部节点，这些节点常常需要一些相应的更新。 对于合约pallet，我们将添加自定义 RPC 端点和初始配置。

添加 RPC API 扩展

在适当的runtime API 暴露后，我们现在可以将RPC 添加到节点服务中来调用 runtime API。 因为我们现运行在外部节点中， 我们没有为 no_std 构建并且我们 没有必要维持专用的 std 功能。

node/Cargo.toml

[dependencies]
jsonrpc-core = '15.0.0'
structopt = '0.3.8'
#--snip--
# *** Add this 2 lines ***
pallet-contracts = '2.0.0'
pallet-contracts-rpc = '0.8.0'

Substrate 提供一个 RPC 与我们的节点交互。 然而，默认情况下它不包含访问合约pallet的权限。 要与这个pallet进行交互，我们必须扩展现有的 RPC 和添加合约pallet的API实现。

node/src/rpc.rs

    /* --snip-- */
    C: Send + Sync + 'static,
    C::Api: substrate_frame_rpc_system::AccountNonceApi<Block, AccountId, Index>,
    /*** Add This Line ***/
    C::Api: pallet_contracts_rpc::ContractsRuntimeApi<Block, AccountId, Balance, BlockNumber>,
    /* --snip-- */
{
    /* --snip-- */

    // Extend this RPC with a custom API by using the following syntax.
    // `YourRpcStruct` should have a reference to a client, which is needed
    // to call into the runtime.
    // `io.extend_with(YourRpcTrait::to_delegate(YourRpcStruct::new(ReferenceToClient, ...)));`

    /*** Add This Block ***/
    io.extend_with(
        ContractsApi::to_delegate(Contracts::new(client.clone()))
    );
    /*** End Added Block ***/
    io
}

初始配置

并非所有pallet都有一个初始配置，但如果你有，你可以使用文档了解它。 例如,pallet_contracts::GenesisConfig 文档 描述了您需要定义的所有Contracts模块字段。

初始配置在 node/src/chain_spec.rs 中控制。 我们需要将此文件修改为包括 ContractsConfig 类型和顶部的合约价格单位：

node/src/chain_spec.rs

Start Your Upgraded Chain

Now you are ready to compile and run your contract-capable node. Compile the node in release mode with

WASM_BUILD_TOOLCHAIN=nightly-2020-10-05 cargo build --release

现在通过运行此命令来启动您刚刚建立的可执行程序

# 在开发模式运行一个临时节点
./target/release/node-template --dev --tmp

Adding Other FRAME pallets

在本教程中，我们专门介绍了如何导入合约模块，但是正如开头所提到的，每个模块的导入方法都会有些许不同。 请不要担心，您始终可以参考演示Substrate node runtime 文档，该runtime几乎包括核心库中的每个模块。

在 Substrate 节点 runtime 的 Cargo.toml 文件中，您将看到如何导入不同的模块，在 lib.rs 文件中，您会看到如何把每个模块添加到您的 runtime 里去。 您基本上可以将在那里完成的操作复制到您自己的runtime。
