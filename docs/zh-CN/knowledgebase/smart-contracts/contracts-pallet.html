合约模块

合约模块为 Runtime 提供了部署和执行 WebAssembly (Wasm) 的能力。

Wasm 引擎

合约模块依赖于 Wasm 的沙盒接口，定义了 Runtime 内可用的 Wasm 执行引擎。 当前基于 Wasm 解释器 wasmi 实现。

功能

合约模块在智能合约的部署和执行上有许多合约开发者熟悉的功能和一些新的功能。

基于帐户

合约模块采用的是类似于现有很多智能合约平台的账户体系。 对 Substrate Runtime 来说，合约账户就像普通的用户账户一样；但是，除了普通账户所拥有的 AccountID 和 Balance 之外，合约账户还有相关的合约代码和一些持久的合约存储。

两步开发

用 Contracts 模块部署合约需要两个步骤：

  1. 在区块链上存储 Wasm 合约。
  2. 用与 Wasm 合约相关的新存储实例化一个新帐户。

这意味着可以使用同一个 Wasm 代码初始化多个具有不同的构造参数的合约实例，从而减少区块链上 Contracts 模块所需的存储空间。

合约调用

对合约的调用可以改变合约的储存、创建新合约，以及调用其他合约。 因为 Substrate 为您提供了编写自定义 Runtime 模块的能力，Contracts 模块也让您可用合约账户直接异步调用那些 Runtime 函数。

沙盒

合同模块旨在供公共网络上的所有用户使用。 这意味着合约只能直接修改他们自己的存储。 为了给底层区块链状态提供安全保障，合约模块实现了可逆交易，通过合约调用回滚没有成功完成的对存储的任何改变。

手续费

合同调用要收取 gas 费用，以限制交易可以使用的计算资源量。 在构造合约交易时，指定了 gas 限额。 随着合约的执行，gas 根据计算的复杂性逐步被消耗。 如果在合约执行完成前达到 gas 限额，则交易失败，合约存储被还原，gas 费用 并不 退还给用户。 如果合约执行完成时还有剩余 gas，则在交易结束时退还给用户。

合约模块决定 gas 的价格，即 Substrate weight system 和单个 gas 间的换算。 因此，要执行一项交易，用户必须有至少 gas price* gas limit 的可用余额，才能消费。

存储租用

类似于 gas 限额对交易所使用的计算资源的限制，存储租金限制了合约在区块链存储中的占用时长。 合约账户按其使用的存储量所占的比例支付租金。 当合约可用余额低于某个限额时，合约账户会变成一个“墓碑”，其存储被清空。 墓碑合约可通过追加维持其活跃所需的资金，并提供变为墓碑合约时所清理的数据来重新激活。

合约模块与 EVM 的关系

合约模块对智能合约生态系统中的现有理念进行了迭代，尤其是 Ethereum 和 EVM。

合约模块和 EVM 之间最明显的区别是用于运行智能合约的底层执行引擎。 EVM 理论上是很好的执行环境，但在使用现代硬件时并不实用。 例如，在现代体系架构上操作256位整数要比标准类型复杂得多得多。 甚至以太坊团队也在为下一代网络调研 Wasm 使用的可能性。

EVM 只在储存发生时收取储存费。 这种一次性的成本导致区块链上的一些存储空间被永久占用，这是很不经济的。 合约模块尝试通过 storage rent 来改进这一点，以确保在区块链上持续存在的任何数据都会因对资源的占用被适当收费。

合约模块选择使用 two-step process 来创建合约，这从根本上改变了合约在链上的存储方式。 合约地址、占用储存和余额现在与基础合约逻辑进行了分离。 这可能会启用类似于以太坊的 create2 的行为，或在基于 Substrate 区块链上启用的可修复或可升级合约。
