Glossary

Глоссарий (алфавитный) терминов, используемый в кодах Substrate и Polkadot, а также архитектуре.

Автор (как Автор блока, Производитель блока)

Автор (иначе автор блока, производитель блока) Узел (нода), агент или личность, ответственные за создание блока. Синоним с "майнером" или "coinbase" на старых PoW цепочках.

Адаптивное смещение кворума (AQB)

Адаптивное смещение кворума (Aqb) Способ определения кворума, необходимого для принятия предложения, зависит от явки избирателей. Положительный сдвиг означает, что требуется все больше голосов «за», чем «против», поскольку явка избирателей ниже; Отрицательное (иначе инвертированное) смещение является противоположным (см. графом). Это позволяет избежать необходимости в строгих кворумах, которые являются произвольными и приводят к нежелательным механикам.

AfG

Внутреннее кодовое имя для "Al's Finality Gadget", который назван в честь Алистера Стюарта, который его изобрел. Смотрите GRANDPA.

Агрегирование

Используемый в контексте "агрегации модулей", это означает объединение аналогичных типов из нескольких модулей среды исполнения в один тип enum с вариантами, позволяющими представлять аналогичный тип каждого модуля. В настоящее время существует пять таких типов данных:

  • Log Журнал (элемент расширенного заголовка)
  • Event Событие (индикатор определенного состояния-перехода)
  • Call Вызов (Функтор (англ. functor, функциональный объект), позволяющий вызывать опубликованную функцию с набором аргументов)
  • Origin Источник (условие вызова функции)
  • Metadata Метаданные (информация, позволяющая проанализировать вышесказанное)

Одобрительное голосование

Одобрительное голосование Система голосования, где избиратель может проголосовать за столько кандидатов, сколько пожелает. Побеждает кандидат с наибольшим общим количеством голосов. Особенности:

  • голосование за всех кандидатов в точности эквивалентно голосованию ни за кого; и
  • можно проголосовать "против" одного кандидата, проголосовав за всех остальных кандидатов.

Авторитеты

Авторитеты - это действующие лица, ключи или личности, которые коллективно управляют консенсусом в сети. AuthorityId может быть использован для их идентификации. В цепи PoS (например, в одной из систем стейкинга SRML) авторитеты определяются с помощью взвешенной по токенами системы номинации/голосования.

Мы иногда используем "авторитеты" и "валидаторы" для обозначения того, что кажется одним и тем же. "Валидаторы" - это более широкий термин, который может включать другие аспекты бслуживания цепи, такие как проверка парачейна. В общем случае авторитеты являются (нестрогим) подмножеством валидаторов, и многие валидаторы будут авторитетами.

Aura (aka "Authority Round")

Детерминированный консенсусный протокол с Не-Мгновенной Финальностью, когда создание блоков производится с помощью чередующегося списка авторитетов, которые поочередно выпускают блоки, и где большинство из них в сети являются честными авторитетами. Смотрите Aura

Aurand

Вариант Aura, где авторитеты случайным образом перемешиваются в каждом раунде, увеличивая безопасность.

Aurand/Ouroboros

Расширение Aurand, где производство блоков включает валидаторов, конкурирующих за ограниченны слоты, которые двигаются очень быстро, где большинство слотов не заполнены, но с редкими коллизиями слотов.

Aurand+GRANDPA

Гибридная схема консенсуса, в которой Aurand используется для производства блоков и краткосрочной вероятностной финальностью, с долгосрочной абсолютной финальностью, обеспечиваемой через GRANDPA.

Слепое назначение расширения блокчейна (BABE)

Слепое назначение расширения блокчейна/Blind Assignment of Blockchain Extension (BABE) Протокол авторизации блоков аналогичен протоколу Aura, в котором авторитеты выбираются с помощью проверяемой случайной функцией (VRF) вместо чередующегося списка. Авторитетам назначается временной слот, в котором они могут выбрать цепочку и отправить для неё новый блок. Смотрите BABE

Блок

Один элемент блокчейна (или парачейна), который объединяет ряд внешних (англ. extrinsic) данных ("тело") вместе с некоторой криптографической информацией ("заголовок"). Блоки упорядочиваются в дерево через родительские указатели (реализованные как криптографические дайджесты родителя), а дерево помещается в список с помощью "правила выбора форка" (англ. fork-choice rule).

Византийская отказоустойчивость (BFT)

Византийская отказоустойчивость (BFT) Возможность распределенной компьютерной системы оставаться работоспособной с учетом доли неисправных узлов или агентов. «Византийский» относится к предельному уровню дефектности, когда предполагается, что такие узлы были заведомо злонамеренными и скоординированными , а не просто находятся в режиме оффлайн или с ошибками. Как правило, системы BFT остаются функциональными примерно с одной третью византийских узлов.

Византийский отказ

Потеря системной службы из-за Византийского Отказа (т.е. компоненты в системе перестали работать и существует несовершенная информация о том, перестал ли компонент работать) в системах, которые требуют консенсуса.

Consensus

Консенсус - это процесс, в котором валидаторы соглашаются на следующий валидный блок, который будет добавлен в цепочку ретрансляции.

Механизм консенсуса

Механизм консенсуса - средство для формирования консенсуса относительно того, что представляет собой (единая истинная) каноническую цепь. В контексте Subtrate он относится к набору реализаций признаков, которые диктуют, какой из ряда входных блоков образует цепочку.

Алгоритмы консенсуса

Алгоритмы консенсуса Алгоритм, который способен гарантировать, что ряд агентов, которые не обязательно доверяют друг другу, могут прийти к консенсусу посредством некоторых вычислений. Часто упоминалось наряду с "безопасностью" (способность обеспечить, чтобы любой прогресс в конечном счете был согласован как случившийся всеми честными узлами) и "живучестью" (способность продолжать добиваться прогресса).

См. GRANDPA Block Finality in Polkadot

Криптографические примитивы

Криптографические примитивы Схема подписи и алгоритм хэширования. Они используются для:

  • Блокчейн: блоки должны быть хэшированы по некоторому алгоритму и должны ссылаться на хэш своего родительского блока.
  • Состояние: Хранилище кодировано как префиксное дерево (англ. trie), предоставляющее криптографическую ссылку на любое заданное состояние, которое использует алгоритм хэширования.
  • Консенсус: авторитеты должны часто использовать какие-либо схемы цифровой подписи.
  • Аутентификация транзакций: в средах исполнения, где используются транзакции и учетные записи, учетные записи должны быть связаны с некоторой цифровой идентификацией.

В SRML, SR25519 и ED25519 предпочитается в ECDSA/SECP256k1.

Council

Совет является одним из примитивов управления SRML и является органом делегатов, выбранным из ряда утверждённых голосов на фиксированные сроки. Совет в первую очередь служит органом, который оптимизирует и проверяет/балансирует интегрированную систему референдума. Он обладает рядом полномочий:

  • представлять референдумы с удаленным или инвертированным (только при единогласном решении) AQB; и
  • отменить референдум (только при единогласном решении); и
  • отказаться от своего места

Движок базы данных

Движёк базы данных Средства, с помощью которых данные, связанные с блокчейном, сохраняются между вызовами узла.

Фраза разработчика

Мнемоническая фраза, которая преднамеренно предается гласности. Все известные учётные записи для разработчиков (Alice, Bob, Charlie, Dave, Ferdie, Eve) генерируются из одной и той же фразы разработчика. Фраза разработчика состоит из: bottom drive obey lake curtain smoke basket hold race lonely fit walk.

Многие инструменты в экосистеме Substrate, такие, как subkey, разрешает пользователям неявно указать фразу разработчика только указав путь деривации, такой как //Alice.

Digest

Дайджест Расширенное поле заголовка блока, которое кодирует информацию, необходимую только для "заголовочных" ("легких", англ. header-only) клиентов для синхронизации цепи.

Dispatch

Отсылка Выполнение функции с заранее определенным набором аргументов. В контексте SRML это относится конкретно к системе "отсылка к среде исполнения", способу получения некоторых чистых данных (тип известен как Call по соглашению) и интерпретации его для вызова опубликованной функции в модуле среды исполнения с некоторыми аргументами. Такие опубликованные функции занимают один дополнительный параметр, известный как origin позволяющий функции надежно определить происхождение её исполнения.

Equivocating

Voting or otherwise backing multiple mutually-exclusive options within the consensus mechanism. Это считается фундаментальным византийским поведением.

Ethash

Один из числа доказательств работы, который используется в алгоритме консенсуса Proof-of-Work (PoW). Первоначально разрабатывался для Ethereum командой, возглавляемой Tim Hughes.

Events

События Средство записи, предназначенные для мира вне цепи (англ. off-chain), что произошел какой-то определенный переход состояния. В контексте SRML события являются одним из нескольких типов данных, со свойствами амальгаммируемости (англ.amalgamatable), которые каждый модуль определяет индивидуально и которые объединены в единый общий enum, который может представлять все типы модуля. События реализуются через переходный набор элементов хранения, которые проверяются сразу после выполнения блока и сбрасываются во время инициализации блока.

Executor

Исполнитель Способ выполнения вызова функции в заданной среде исполнения с набором внешних факторов/данных. There are two executor implementations present in Substrate, Wasm and Native.

Wasm исполнитель

Исполнитель, который использует бинарный Wasm и интерпретатор Wasm для выполнения необходимого вызова. Это обычно медленно, но гарантировано правильно.

Встроенный исполнитель

Исполнитель, который использует текущую встроенную и нативно (англ. native) скомпилированную среду исполнения для выполнения необходимого вызова. Если код совместим с кодом Wasm в цепи (англ. on-chain), то это будет намного быстрее и правильнее. Неправильное управление версиями приведет к ошибке консенсуса.

Extrinsic

A piece of data bundled into a block that expresses something from the "external" (i.e. off-chain) world. There are, broadly speaking, two types of extrinsic: transactions (which tend to be signed) and inherents (which don't).

Existential Deposit

Экзистенциальный депозит В модуле SRML Balances это минимальный баланс, который может иметь счет. Счета не могут быть созданы с балансом меньше некоторой минимальной суммы, и если баланс счета когда-либо падает ниже суммы, то он полностью удаляется из системы.

Пока и отправитель, и получатель сохраняют баланс хотя бы экзистенциального депозита, переводы, в том числе и очень небольших сумм, будут работать интуитивно. Однако попытка перевести небольшой баланс с одного счета на другой может иметь несколько менее интуитивные последствия, когда баланс отправителя или получателя очень низок.

Если переведенная сумма меньше экзистенциального депозита, а целевой счет ранее не существовал, то перевод будет "успешным" без фактического создания и кредитования целевого счета; это, по-видимому, просто сжигает баланс перевода от отправителя. Если передача приведет отправителя к значению ниже экзистенциального баланса, то его учетная запись будет удалена. Таким образом, передача может даже "успешно" привести к тому, что учетная запись отправителя будет полностью удалена, а учетная запись получателя никогда не будет создана.

Это промежуточный слой программного обеспечения, чтобы гарантировать, что конечные пользователи осведомлены и/или защищены от этих крайних случаев.

Finality

Финальность Часть консенсуса, связанная с тем, чтобы сделать прогресс необратимым. Если блок будет финализирован, то любые реальные последствия могут быть выполнены. Алгоритм консенсуса должен гарантировать, что финализированные блоки никогда не будут обращены.

Один из примеров доказуемого финализирующего гаджета используется в Polkadot: GRANDPA.

См. также: Aurand+GRANDPA, Instant Finality, Proof-of-Finality, Probabilistic Finality, Provable Finality

Full Client

Полный клиент Узел, способный синхронизировать цепочку блоков максимально безопасным образом посредством выполнения (и, следовательно, проверки) всей логики. Сравните с Light Client.

Genesis Configuration

A JSON-based configuration file that can be used to determine a genesis block and which thus allows a single blockchain runtime to underpin multiple independent chains. Analogous to a "chain spec" file in Parity Ethereum, and, when used with Substrate Node, could be considered the third and highest-level usage paradigm of Substrate. SRML обеспечивает средство автоматического генерирования логики конфигурации цепи из модулей и их постоянных элементов хранения.

GRANDPA

GRANDPA (GHOST-based Recursive ANcestor Deriving Prefix Agreement), also previously known as SHAFT (SHared Ancestry Finality Tool), and originally known as AfG, is a finality gadget for blockchains, implemented in Rust. Официальная спецификация опубликована на grandpa.pdf.

Header

Заголовок Части, в первую очередь криптографической информации, которая суммирует блок. Эта информация используется лёгкими клиентами для получения минимально безопасной, но очень эффективной синхронизации цепи.

HoneyBadgerBFT

Instant-finality consensus algorithm with asynchronous "safety" and asynchronous "liveness" if tweaked. Originally slated as one of a number of directions that Polkadot might explore, but unlikely to be used now.

Hybrid Consensus Protocol

Split blockchain consensus into Block Production and a Finality Gadget. This allows chain growth speed to be as fast as in probabilistic "safety" consensus such as Ouroboros or Aurand but with same level of security guarantees as in Instant-Finality Consensus Protocols. See the first implementation in Rust, and the GRANDPA article.

Instant-Finality

A non-probabilistic consensus protocol that gives a guarantee of finality immediately upon block production, for example Tendermint and Rhododendron. These tend to be PBFT based and thus very expensive in terms of communication requirements.

JSON-RPC

Стандартный вызов функций на удаленной системе с помощью протокола JSON. Для Substrate это осуществляется через JSONRPC Parity crate.

JSON-RPC Core Crate

Позволяет создавать обработчик JSON-RPC сервера с зарегистрированными поддерживаемыми методами. Предоставляет ядро субстрата через различные типы транспортных протоколов (например, WS, HTTP, TCP, IPC)

JSON-RPC Macros Crate

Allows simplifying in code the creation process of the JSON-RPC server through creating a Rust Trait that is annotated with RPC method names, so you can just implement the methods names

JSON-RPC Proxy Crate

Представляет простой сервер, такой как TCP из бинарных файлов, с другим двоичным файлом на фронте, чтобы служить прокси-сервером, который будет предоставлять все другие транспортные протоколы, и обрабатывать входящие вызовы RPC прежде чем они поступят на вышестоящий сервер, что позволяет реализовать промежуточное ПО кэширования (сохраняет необходимость пройти весь путь до узла), разрешает промежуточное ПО для балансировки нагрузки между экземплярами узла или перемещение управления учетными записями на прокси-сервер, который обрабатывает подписанную транзакцию. Это дает альтернативу встраиванию всего JSON-RPC в каждый проект и всех параметров конфигурации для каждого сервера.

JSON-RPC PubSub Crate

Пользовательское (хотя и обычное) расширение, которое полезно для разработчиков Dapp. Он формирует "подписки" (англ. subscriptions), так что сервер отправляет уведомления клиенту автоматически (вместо того, чтобы вызывать и опрашивать удаленную процедуру вручную все время), но только с транспортными протоколами, которые поддерживают постоянное соединение между клиентом и сервером (т. е. WS, TCP, IPC)

Keystore

A subsystem in Substrate for managing keys for the purpose of producing new blocks.

Libp2p Rust

Одноранговая сетевая библиотека. Based on Protocol Labs' Libp2p implementations in Go and JavaScript, and their (somewhat incomplete) specifications. Allows use of many transport protocols including WebSockets (usable in a web browser).

Metadata

Information capture allowing external code to introspect runtime data structures. Includes comprehensive descriptions of the dispatch functions, events and storage items for each module in the runtime.

Nominated Proof-of-Stake (NPoS)

A means of determining a set of validators (and thus authorities) from a number of accounts willing to commit their stake to the proper (non-Byzantine) functioning of one or more authoring/validator nodes. Это было первоначально предложено в документе Polkadot и формулирует набор номинаций, на которые делают ставку, как проблему оптимизации ограничений, чтобы в конечном итоге дать набор валидаторов, на который максимально делают ставку, каждый из которых имеет ряд поддерживающих номинаторов, предоставляющих свою долю. Сокращения и вознаграждения делаются пропорционально.

OAS3

Спецификация OpenAPI 3 (OAS3), которая официально называлась спецификацией Swagger API, является спецификацией, используемой для создания файла Swagger в формате JSON или YAML, который может использоваться для создания документации по API.

Origin

Источник. Происхождение отправленного вызова функции в среде исполнения. Может быть настроен (например, Совет источник "Движения"), но существует два специальных "встроенных":

  • Корневой (англ. Root: уровень происхождения - системный, который, как предполагается является всемогущим;
  • Подпись: источник - транзакция, включает идентификатор учетной записи подписывающего лица;

Practical Byzantine Fault Tolerance (pBFT)

Оригинальный метод для решения проблем византийских генералов. Это позволяет обеспечить устойчивость системы к византийскому поведению от одной трети её участников сO(2 N**2) коммуникационными накладными расходами на узел.

Probabilistic Finality

В цепочке, основанной на вероятностной финальности (например, биткойн), участники сети полагаются на некоторую вероятность p, что предлагаемый блок B останется в канонической цепочке бесконечно, причем p приближается к 1, поскольку дополнительные блоки создаются поверх блока B.

Proof-of-Finality

Доказательство финальности. Часть данных, которая может быть использована для доказательства того, что конкретный блок завершен. Потенциально очень большой, если не используется агрегация сигнатур.

Provable Finality

Некоторые механизмы консенсуса нацелены на доказуемую финальность, при которой все блоки гарантированно являются каноническим блоком для этой цепи при включении блока. Доказуемая финальность желательна в тех случаях, когда легкие клиенты не имеют полного состояния цепи, или при взаимодействии с другими цепями, где связаные данные цепей не распределены повсеместно. См.: GRANDPA

Rhododendron

Алгоритм консенсуса с Мгновенным Финальным BFT, один из ряда адаптаций PBFT для блокчейна. Другие включают в себя алгоритм консенсуса Stellar и Tendermint. См. Rhododendron crate.

Runtime

The block execution logic of a blockchain, i.e. the state transition function. В Substrate это хранится в цепочке в независимом от реализации машинно-исполняемом формате в виде двоичного файла WebAssembly. Other systems tend to express it only in human-readable format (e.g. Ethereum) or not at all (e.g. Bitcoin).

SHAFT

SHAFT (SHared Ancestry Finality Tool). См.: GRANDPA.

Stake-Weighted Voting

Democratic system with one-vote-per-token, rather than one-vote-per-head.

State

The data that can be drawn upon by, and that persists between, the executions of sequential blocks. State is stored in a "Trie", a cryptographic immutable data-structure that makes incremental digests very efficient. In Substrate, this trie is exposed to the runtime as a simple key/value map where both keys and values can be arbitrary byte arrays.

SRML (Substrate Runtime Module Library)

An extensible, generic, configurable and modular system for constructing runtimes and sharing reusable components of them. Broadly speaking, this corresponds to the second usage paradigm of Substrate, higher-level than using Substrate Core, but lower-level than simply running Substrate Node with a custom configuration.

STF (State Transition Function)

The logic of a blockchain that determines how the state changes when any given block is executed. In Substrate, this is essentially equivalent to the Runtime.

Storage Items

Within the SRML, storage items are a means of providing type-safe persistent data for the runtime, achieved with a compiler macro, the parity-codec crate and the State API. All storage items must have a type for which the parity-codec::Codec trait is implemented, and, if they have a corresponding JSON Genesis Configuration entry, then also the serde traits.

Substrate

A framework and toolkit for building and deploying upgradable, modular and efficient blockchains. There are three usage paradigms in increasing levels of functionality and opinionation: Core, SRML and Node.

Ядро Substrate

The lowest level of the three Substrate usage paradigms, this is a minimalist/purist blockchain building framework that contains essential functionality at the lowest level including consensus, block production, chain synchronisation, I/O for JSON-RPC, runtime, network synchronisation, database backend, telemetry, sandboxing, and versioning.

Окружение выполнения Substrate

The runtime environment under which WebAssembly code runs in order to execute blocks.

Transaction

A type of Extrinsic that includes a signature, and all valid instances cost the signer some amount of tokens when included on-chain. Because validity can be determined efficiently, transactions can be gossipped on the network with reasonable safety against DoS attacks, much as with Bitcoin and Ethereum.

Transaction Era

A definable period, expressed as a range of block numbers, where a transaction may validly be included in a block. Eras are a backstop against transaction replay attacks in the case that an account is reaped and its (replay-protecting) nonce is reset to zero. Eras are efficiently expressible in transactions and cost only two bytes.

See reclaiming an account.

Transaction Pool

A collection of transactions that are not yet included in blocks but have been determined to be valid.

Пул транзакций с тегом

A "generic" transaction pool implementation allowing the runtime to specify whether a given transaction is valid, how it should be prioritised and how it relates to other transactions in terms of dependency and mutual exclusivity. It is designed to be easily extensible and general enough to have both UTXO and account-based transaction modules be trivially expressible.

Trie (Patricia Merkle Tree)

An immutable cryptographic data-structure typically used to express maps or sets of items where:

  • необходим криптографический дайджест набора данных; и/или
  • дешево пересчитать дайджест с поэтапными изменениями в наборе данных, даже когда он очень большой; и/или
  • требуется краткое доказательство того, что набор данных содержит некоторый предмет/пару (или его отсутствие).

Validator

A semi-trusted (or untrusted but well-incentivised) actor that helps maintain the network. In Substrate, validators broadly correspond to the authorities running the consensus system. In Polkadot, validators also manage other duties such as guaranteeing data availability and validating parachain candidate blocks.

See https://wiki.parity.io/Validator-Set.html

WebAssembly (Wasm)

An execution architecture based upon a virtual machine that allows for the efficient, platform neutral expression of deterministic machine-executable logic. Wasm is used across the Web platform and well-tooled and easily compiled from Rust.
